PagedAttention: Virtual Memory for LLM Serving

Efficient Memory Management for Large Language Model Serving with PagedAttention

📅 SOSP 2023 👤 W. Kwon et al. (vLLM Team, UC Berkeley) 🏢 UC Berkeley, Stanford
📄 arXiv:2309.06180 📑 PDF 💻 vLLM Code

🎯 The Problem: KV Cache Memory Waste

Large Language Models generate tokens autoregressively, requiring the storage of Key and Value tensors (KV cache) from all previous tokens. This creates a massive memory bottleneck:

The Memory Explosion: For a 13B parameter model with batch size 64 and sequence length 4096, the KV cache alone requires ~160GB of GPU memory—more than the model parameters themselves!

Existing systems use contiguous memory allocation for KV cache, which leads to three major inefficiencies:

  • Internal Fragmentation: Pre-allocate max sequence length, but most sequences are much shorter
  • External Fragmentation: Different sequence lengths leave unusable memory gaps
  • No Memory Sharing: Beam search and parallel sampling duplicate KV cache
Figure 1: Contiguous vs Paged Allocation

💡 The Innovation: Virtual Memory for KV Cache

PagedAttention draws inspiration from virtual memory in operating systems. Just like OSes use pages and page tables to map virtual addresses to physical memory, PagedAttention uses:

1. Fixed-size Blocks: KV cache is divided into fixed-size blocks (e.g., 16 tokens per block)
2. Block Tables: Each sequence has a block table mapping logical token positions to physical blocks
3. Non-contiguous Storage: Logical blocks can map to any physical blocks, eliminating contiguous allocation requirements

The block table for each sequence stores the mapping from logical block indices to physical block indices. This simple abstraction enables powerful optimizations:

Figure 2: Block Table Structure

🔄 Memory Sharing: Copy-on-Write

PagedAttention enables powerful memory sharing techniques through copy-on-write (COW) semantics:

Beam Search: Multiple candidate sequences share common prefix blocks. When a candidate diverges, only then is a new physical block allocated.

Parallel Sampling: Multiple outputs from the same prompt share all initial blocks, dramatically reducing memory for sampling.

💻 Using vLLM

Using vLLM with PagedAttention is straightforward:

from vllm import LLM, SamplingParams # Load a model - PagedAttention is used automatically! llm = LLM(model="meta-llama/Llama-2-7b-hf") # Define sampling parameters sampling_params = SamplingParams( temperature=0.8, top_p=0.95, max_tokens=256 ) # Batch inference - PagedAttention handles memory automatically prompts = [ "The future of AI is", "In the year 2050,", "The most important invention" ] # Generate - PagedAttention manages KV cache efficiently outputs = llm.generate(prompts, sampling_params) # PagedAttention benefits: # - No manual batch size tuning # - Automatic memory management # - Support for beam search with COW # - Continuous batching for output in outputs: prompt = output.prompt generated_text = output.outputs[0].text print(f"Prompt: {prompt!r}") print(f"Generated: {generated_text!r}")

For production serving with OpenAI-compatible API:

# Start vLLM server with PagedAttention # Command line: python -m vllm.entrypoints.openai.api_server \ --model meta-llama/Llama-2-7b-hf \ --tensor-parallel-size 1 \ --max-num-seqs 256 \ --max-model-len 4096 # Client code: import openai client = openai.OpenAI( base_url="http://localhost:8000/v1", api_key="dummy" ) response = client.chat.completions.create( model="meta-llama/Llama-2-7b-hf", messages=[{"role": "user", "content": "Hello!"}] ) print(response.choices[0].message.content)

📝 Summary

PagedAttention revolutionized LLM serving by applying virtual memory concepts to KV cache management:

  • Block-based allocation: Fixed-size pages eliminate fragmentation
  • Block tables: Map logical to physical blocks for flexibility
  • Copy-on-write: Enable memory sharing for beam search and sampling
  • 96% memory utilization: vs 40% for contiguous allocation
  • 2-4× throughput: improvements in production serving
Bottom Line: PagedAttention is the foundational technology behind vLLM, now one of the most popular LLM serving frameworks. By treating GPU memory like virtual memory with paging and copy-on-write, it solved the KV cache memory bottleneck that plagued LLM deployment.
